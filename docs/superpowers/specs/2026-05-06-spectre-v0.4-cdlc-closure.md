## 1. Hard Problem

Spec authorship is the load-bearing irreducibly-human cognitive function (per the essay). But **writing** a 10-page spec by hand is friction the human shouldn't bear — it's transcription work, not authorship work. The author's actual job is **intent + arbitration**, not keystrokes. Today's `/vision` collapses authorship and transcription into one prompt: user types a one-liner, agent drafts a full spec, asks 2-3 surface questions, locks. The result is shallow specs that bypass the asymptotic-completeness walk the essay names as the leverage point.

v0.4 separates the two layers: the LLM walks the possibility-graph (the work biology lets humans skip), the human supplies intent + arbitrates branches. The output is a complete-enough spec without the human keystroking it. Spectre simultaneously closes the rest of the Context Development Lifecycle (Distribute, Observe, Adapt) so that knowledge accumulated in one walk improves every future walk — automatically, invisibly, with author-arbitrated adoption.

## 2. First Principles

- **Cognition is context-bound.** Per the essay, completeness is task-relative and asymptotic; the technical stopping rule is *marginal spec yield* — stop when adding a branch no longer shifts output behavior beyond a tolerance. The human supplies the social stopping signal (deadline, "good enough"); the system supplies the technical one.
- **Spec authorship ≠ spec transcription.** Authorship is intent + arbitration + edge-case enumeration. Transcription is keystrokes. v0.4 keeps the first with the human and moves the second to the LLM.
- **Specs are receiver-relative and multi-receiver.** A spec is read by Claude `/implement` (executor), DeepSeek Tier 3 (adversarial reviewer), the human (post-lock auditor), and deterministic checkers (spec_ast, coverage_gate, spec_lint). Different receivers need different completeness in different sections.
- **Prose-only defenses fail.** Issue #1's recurring failure mode (six gaps closed in v0.3.1) was defenses documented in skill prose with no `bin/` enforcement. v0.4's interrogation walker MUST live in code; the skill renders only.
- **The compliance gap is real.** Per `research/programmatic-vs-reasoning-constraints`, prompt-based rules hit 70-85% compliance; programmatic constraints hit 100%. The walker's state machine is programmatic; only the question rendering is probabilistic.
- **The Sandbox Paradox is real.** Per `research/developing-a-safe`, HITL approval gates create permission-fatigue, leading users to rage-bypass safety. v0.4's live-opt-out flow MUST have a brake to prevent this exact mode.
- **CDLC closure compounds.** Per Debois, context is code; it needs Generate→Test→Distribute→Observe→Adapt. Spectre owns Generate+Test today; v0.4 closes the loop so spec quality improves *between* specs, not just *within* one.

## 3. Algorithm Audit

**Delete:** today's `/vision` flow of "draft full spec from one-liner, ask 2-3 surface questions, lock." Replaced wholesale.

**Simplify:** the walker emits a single primitive — *next unresolved concern* — typed as `{id, kind, depends_on: list[id], receivers: list[str], summary: str}`. The skill phrases this into a natural-language question; the user answers; the walker updates state. No question-bank, no decision tree DSL — the walker holds a graph, the LLM does the NL phrasing.

**Accelerate:** parallel receiver-stratified questioning. The walker can queue questions for `/implement` receiver and `Tier-3` receiver simultaneously; skill picks one to render per turn. Author can switch receivers mid-walk if a strong intent signal emerges.

## 4. Speed-of-Light Limit

A walk for a moderately complex spec (~15 step targets) takes ~10-15 minutes of author time at 1 question per ~30 seconds. Without the walker, the same spec depth costs ~60-90 minutes of hand-authoring plus a 30-second Tier 3 pass that catches half what the walker would surface live. v0.4 is ~6x faster to a deeper spec, with the walker's emit-time receiver stratification doing work that today happens only post-hoc in Tier 3.

## 5. Physics Guardrails

- **Walker state is durable.** `state/.walk.json` (per-spec, per-track, atomically written) survives session interruption. Resume re-reads state, replays last unanswered question.
- **Author override always wins.** Stopping is socially negotiated *above* technical yield. The author can stop at any time regardless of `pending_questions` length.
- **Sandbox-paradox brake.** Live opt-out adoptions cap at 3 per session before requiring `/spectre review-rules` (single-shot acknowledgment). Prevents permission-fatigue rage-adoption.
- **Receiver translation is named-fragile.** The walker's emitted concern is canonical structured data. The skill's question-rendering is best-effort. The user is reading questions; debugging means reading the structured concern in `state/.walk.json`.
- **Personal rules can never weaken project-locked rules.** A spec's `## 8.1 Hard contract` always overrides personal-rules.toml. Personal rules only apply to global tier classifications, never to spec-declared mutates/never-touches.
- **Removal is manual.** No "remove this rule?" prompt. Personal rules accumulate; pruning is an explicit author action against the file.

## 6. Components

### 6.1 `bin/walker.py` (NEW) — strict-hybrid state machine

**Responsibility:** owns walk state, branching, stop conditions, invalidation. Emits the next unresolved concern. Does NOT phrase questions or capture user input.

**Public API:**
```python
@dataclass
class Concern:
    id: str
    kind: str  # "edge-case" | "receiver-clarification" | "assumption-surface" | "branch-resolution"
    receivers: list[str]  # ["implement", "tier3", "human", "deterministic"]
    depends_on: list[str]  # parent concern ids
    summary: str  # canonical structured description

@dataclass
class WalkState:
    spec_intent: str  # the original one-liner from the human
    spec_draft_path: pathlib.Path  # accumulated draft as it materializes
    asked: list[Concern]  # concerns surfaced to user
    answered: dict[str, str]  # concern_id → user answer
    pending: list[Concern]  # concerns queued
    stale: set[str]  # concerns invalidated by author revision
    stop_reason: str | None  # "author-arbitrated" | "tier3-yield-converged" | "max-rounds" | "per-receiver-exhausted" | None

def init_walk(spec_intent: str, spec_path: pathlib.Path) -> WalkState
def next_concern(state: WalkState) -> Concern | None  # returns None if walk should stop
def record_answer(state: WalkState, concern_id: str, answer: str) -> WalkState  # may add new concerns + mark dependents stale
def revise_answer(state: WalkState, concern_id: str, new_answer: str) -> tuple[WalkState, list[str]]  # returns (new_state, invalidated_concern_ids)
def should_stop(state: WalkState) -> tuple[bool, str | None]  # checks all stop conditions
def persist(state: WalkState, path: pathlib.Path) -> None
def load(path: pathlib.Path) -> WalkState | None
```

**Stop conditions (any triggers halt):**
1. Author types `stop` / `lock` at any point.
2. Tier 3 finding-set delta drops below threshold (3 consecutive rounds with <2 new fingerprints).
3. Max-rounds hit (default 30; configurable via `~/.spectre/walker.toml`).
4. Per-receiver exhaustion (no concerns remain for any receiver).

**Invalidation semantics:** dependency-tracked. When `revise_answer` is called, walker computes transitive closure of `depends_on` graph and marks downstream concerns `stale`. Skill renders the diff to the user; user picks `re-walk` or `accept-stale`.

### 6.2 `skills/vision/SKILL.md` (HEAVILY MODIFIED) — thin renderer

**v0.3.x §6.4-§6.7 evaluator pipeline preserved unchanged.** What changes is §1-§5 (intent capture + draft refinement). New flow:

1. **Step 1 (Intent capture):** user types `/vision <one-liner>`. Skill calls `walker.init_walk(intent, draft_path)`.
2. **Step 2 (Walk loop):** skill calls `walker.next_concern(state)`. If non-None, skill phrases the concern's `summary` into a natural-language question, presents to user, captures answer, calls `walker.record_answer`. If None (`should_stop` true), proceed to Step 3.
3. **Step 3 (Materialize draft):** walker has accumulated answers; skill renders the draft from accumulated concerns + answers (what was today the "draft full spec" step, but now grounded in walk-derived state).
4. **Step 4 (Confirmation):** as today — `yes / refine / cancel`. On `refine`, route to walker.revise_answer for affected concerns.
5. **Steps 5+:** unchanged from v0.3.x (§6.4 evaluator → §6.5 ADR → §6.6 Resource → §6.7 lock).

### 6.3 `bin/observations.py` (NEW) — Observe-leg ingester

**Responsibility:** every `/implement` halt, verification fail, tier-gate trigger emits a fingerprinted observation to `~/.spectre/observations.jsonl`. Append-only across all projects.

**Public API:**
```python
def record_halt(*, kind: str, fingerprint: str, project_path: str, spec_slug: str | None, action: str | None) -> None
def find_recurrences(*, kind: str | None = None, threshold: int = 3) -> list[Observation]
def fingerprint_halt(action: str, classifier_label: str) -> str  # SHA256 of normalized (action, label)
```

### 6.4 `bin/personal_rules.py` (NEW) — Adapt-leg ruleset

**Responsibility:** loads `~/.spectre/personal-rules.toml`. Provides predicate functions that `bin/tier.py` and `bin/spec_evaluator.py` consult to skip halts the user has explicitly adopted as not-block-worthy.

**Public API:**
```python
def load_personal_rules() -> PersonalRules  # parsed TOML
def is_classifier_halt_overridden(*, classifier_label: str, fingerprint: str) -> bool
def append_adoption(*, classifier_label: str, fingerprint: str, reason: str) -> None  # writes back
def adoption_count_this_session() -> int  # for sandbox-paradox brake
```

### 6.5 `bin/cdlc_ledger.py` (NEW) — single-source-of-truth audit

**Responsibility:** `state/cdlc-ledger.json` (per-project) records every Generate→Test→Lock→Implement→Halt→Adapt transition with timestamps and fingerprints. Read-only audit surface. No user-facing command — the user reads the file.

### 6.7 Base ruleset categories (`~/.spectre/base-rules.toml`)

Eleven categories, each fingerprinted and named. User edits before first `/vision` to enable/disable. Categories:

1. Engineering best practices (atomic writes, no torn state, pragma test-gaming guard)
2. Software security (OWASP-shaped: cred exfil, untrusted input, injection patterns, path traversal, deserialization)
3. System security (privilege escalation, persistent daemon installation, kernel param mutation, firewall/network rules, user/group management)
4. Supply chain integrity (dependency-add detection, lockfile drift, unverified-binary execution)
5. Data hygiene (PII patterns, secret patterns, egress destinations, telemetry endpoints)
6. Compute boundaries (paid API budgets, rate limits, network egress targets, disk/memory ceilings)
7. Reproducibility (non-deterministic actions, missing version pins, missing `--no-network` for sandboxed builds)
8. Reversibility / blast radius (destructive deletes, force-pushes, schema migrations, mass file moves)
9. Spec-author hygiene (runuser-no-cd, unsafe-heredoc, soft verifications, missing why, missing receiver calibration) — already shipped
10. Resource-lock discipline (declared shared resources, port collisions, file-lock contention) — already shipped
11. Receiver calibration (assumes, runtime-flavor, expected-author-skill) — already shipped

Categories 1, 9-11 partially exist today; v0.4 formalizes them under base-rules.toml. Categories 2-8 are mostly new surface.

### 8.2 Human-facing notes

## 9. Risks & explicit named-fragile surfaces

- **NL question-rendering is the unreliable layer.** Walker emits structured concerns (canonical); skill phrases them (best-effort). When a user complains "this question made no sense," debug by reading `state/.walk.json` for the original concern, not the rendered text. Documented as a known asymmetry, not fixed in v0.4.
- **Tier 3 yield-delta is a proxy for the real measurand.** The essay specifies "shifts the output distribution" — strictly we'd need to *simulate* `/implement` to measure yield. Tier 3 finding-set delta is cheaper and principled (T3 is itself simulating an adversarial receiver). Documented as a proxy.
- **Sandbox-paradox vector.** Live opt-out flow can become a permission-bypass attack surface if user adopts everything. Brake at 3 adoptions/session + manual-only removal limits the damage; not a complete defense. Personal rules cannot weaken project-locked `## 8.1` rules.
- **CDLC ledger is per-project, observations are per-user.** A single user's walks across 10 projects produce 10 ledgers + 1 unified observations.jsonl. Adapt's recurrence detection runs on the unified file (catches cross-project patterns). Documented so users understand why the ledger and observations live in different places.

## 10. Rollout plan

v0.4 is large enough that a single PR is the wrong shape. Decompose into **three releases**:

- **v0.4.0** — `bin/walker.py` + skill rewrite. The interrogation flow only. Distribute/Observe/Adapt are placeholders; ledger writes happen but no Adapt-prompts fire yet.
- **v0.4.1** — `bin/observations.py` + `bin/personal_rules.py` + live-opt-out flow at `/implement` halt sites. Sandbox-paradox brake. Adapt threshold-prompts (≥3× recurrence) fire.
- **v0.4.2** — `bin/cdlc_ledger.py` formalized + Distribute leg (`~/.spectre/templates/` directory + import/export tooling). Adapt's auto-template-patch proposal (when a recurrence shifts the spec_template).

Each release ships independently; v0.4 work targets v0.4.0 first.

## 11. Out of scope (deferred to v0.5+)

- **Multi-user shared rule registries.** A team could share `personal-rules.toml` via git, but v0.4 doesn't engineer for this. v0.5 candidate.
- **Walker-driven auto-fix from Tier 3 findings.** Walker only emits and records; doesn't propose corrections. v0.5+ candidate.
- **Real `/implement`-simulation as marginal-spec-yield measurand.** Replacing the Tier 3 proxy with actual execution simulation is a research-grade problem. Not in scope.
- **Local-LLM walker.** Per CLAUDE.md user policy (thermal incident 2026-04-11), no local LLM inference for Spectre. Walker calls remote DeepSeek for yield-delta only.
