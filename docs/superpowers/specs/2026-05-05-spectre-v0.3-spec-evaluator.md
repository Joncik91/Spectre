# Spectre v0.3 — Pre-Lock Spec Evaluator (CDLC Evaluate Phase)

**Date:** 2026-05-05
**Status:** Architecture brief (pre-implementation). v0.3 Plan A; v0.3 may carry one or more plans (B/C TBD). Test-gaming defense is **deferred to v0.4** and explicitly out of scope here.
**Predecessor:** `2026-05-05-spectre-v2-graph-orchestrator.md` (v0.2.x shipped).

## Why v0.3

v0.2.2 first-production E2E (BTC poller spec, 2026-05-05) shipped successfully but exposed the same failure mode the [Context as Cognitive Substrate](../../../../../root/claude-obsidian/concepts/context-as-cognitive-substrate.md) essay names: **the spec author and the spec executor have overlapping blind spots when they're the same model.**

Real failures observed during that single 7-step run, all caught by the human at execution time and not by any structural mechanism:

1. `/vision` drafted the spec without `resources:` declarations on 4 of 7 steps that bound or scraped a TCP port. The resource extractor only matched `127.0.0.1:8765` literal patterns, not `PORT = 8765` heredoc-embedded constants or `systemctl enable --now <unit>` indirect binders.
2. The chosen port (8765) was already held by the rule-router daemon. Neither `/vision` nor the supervisor had any mechanism to detect external port collisions before the smoke test failed at Step 3.
3. `/vision` placed Resource node generation at §6.6 — between user-confirmation and lock — meaning `resources:` declarations were invisible to the reviewing human at draft-review time.
4. The tier classifier at runtime classified `systemctl daemon-reload` as `silent` (no path captures) and `systemctl enable --now btc-poller.service` as `repo` (matched `.service` as a project file). Both required human-judgment override at execute time.
5. The spec contained an implicit decision (`/var/log/btc-prices/` over `/tmp/btc-prices/` because A8's /tmp is tmpfs) that was articulated only after the human flagged it during refinement. ADR generation fired correctly post-decision but never *prompted* the spec author to surface the constraint.

Every one of these is a **context completeness failure**, not an execution failure. Per the essay:

> **Most failure attributed to "the engine being dumb" is in fact a failure of context handoff between engines.**

CDLC (Patrick Debois) names the same gap: the industry is hyper-focused on the **Generate** phase (prompting). Spectre's `/vision` is firmly in Generate. **Evaluate, Distribute, Observe, Adapt** are missing. v0.3 ships Evaluate.

## The Eight Locked Decisions

### 1. Pre-lock evaluator runs over a *review bundle*, not the raw draft

The evaluator fires inside `/vision` after **Step 6 (draft-to-disk)** but it operates on a **review bundle**, not on the bare draft. The bundle is a transient snapshot containing:

- The `.draft` spec body
- **Preview ADRs** — generated from the draft's decision markers (the §6.5 ADR-generation logic runs in *preview mode*, producing ADR text without writing to `decisions/`)
- **Preview Resource nodes** — `bin.resources.extract_resources_from_action` runs on every step's action; the inferred Resource set is computed without writing to `specs/.graph.md`
- **Preview tier classifications** — every action is run through `bin.tier.classify` ahead of time; the (tier, reasons, never_autonomous) tuple per step is part of the bundle

The evaluator checks the bundle as a unit. After the user replies `yes`, the *real* §6.5 ADR write and §6.6 Resource node write proceed using the same artifacts the evaluator already validated. After the user replies `refine`, the bundle is rebuilt against the refined draft.

**Why a bundle, not the bare draft:** the previous brief had a structural inconsistency (Copilot review caught it 2026-05-05): Tier 1 was supposed to check "decision without ADR" and "undeclared resource," but the evaluator ran *before* ADR generation and Resource node generation. That order is incoherent — the evaluator can't validate artifacts that don't exist yet. The bundle materializes preview artifacts so the evaluator has something to validate, and the *commit* step (real ADR write + Resource node write) only happens after `yes`.

**Sleeper risk:** evaluator latency adds friction. Tier 1+2 together must stay <2s; Tier 3 must stay <30s. Above that, users skip refinement.

### 2. Three-tier architecture (Pragma pattern, adapted)

Borrowed structurally from `apps/Pragma/`'s test-gaming detector. **Pragma's target is test→code coverage; Spectre v0.3's target is spec→action completeness.** The architecture is the same:

| Tier | Cost | Always-on? | What it catches |
|---|---|---|---|
| **Tier 1 — AST/structural classifier** | ~10ms, deterministic | yes | tautological verifications (`echo done`, `true`, `[ 1 -eq 1 ]`); missing `why:`; action-effect not probed by verification; undeclared host paths from `bin/tier.classify`; undeclared ports from `bin.resources.extract_resources_from_action`; decision markers in §2 that didn't generate ADRs |
| **Tier 2 — coverage-of-action gate** | ~1s, deterministic | **always-on (changed from opt-in per Copilot review 2026-05-05)** | dry-run-classify every action, then check the spec's `resources:` declarations cover the inferred set; verification command must structurally probe at least one path that the action wrote (file existence check, content grep, exit code of a derived process) |
| **Tier 3 — DeepSeek v4 Pro adversarial reviewer** | ~10-30s, ~$0.01-0.05 per call | opt-in | three-prompt structured probing: (a) what the spec doesn't say, (b) what the spec asserts that's wrong, (c) attacker view |

**Why three tiers, not one DeepSeek call:** marginal cost. Tier 1 catches the easy 80% deterministically and free. Tier 2 catches the structural-coverage 15% with a dry-run. Tier 3 spends real money/latency on the residual 5% that requires genuine model reasoning. Layered defense, not stacked.

**Default tier policy:** Tiers 1 + 2 are baseline (always run; both are local + deterministic). Only Tier 3 (paid network call to DeepSeek) is opt-in. Skipping Tier 2 by default would have caused the v0.2.2 BTC poller spec to lock without the missing-resources finding — exactly the failure this brief exists to prevent.

### 3. Reviewing model is structurally other-than-Claude

Per the essay's adversarial-reviewer principle:

> The spec author is also a cognitive node with blind spots; the reviewer is the structural answer to "who watches the walker?" Without this role, the spec author's blind spots become the system's blind spots.

Spec author = Claude Code via `/vision`. Reviewing model for Tier 3 = **DeepSeek v4 Pro** via OpenAI-compatible API. Different training distribution, genuinely different blind spots. Same-family review (another Claude) does not satisfy the principle.

**Sleeper risk:** the API key represents a paid network call (`Never Autonomous` per `bin/tier.py`). Must be opt-in via `~/.spectre/reviewer.toml`. Default behavior = Tier 1 only. Tier 3 fires only when the user has explicitly authored the config.

### 4. Findings are JSON, not narrative

Every finding the evaluator produces is a typed dict. Locations support spec-wide, cross-step, and per-step findings:

```json
{
  "tier": 1 | 2 | 3,
  "kind": "missing-why" | "soft-verification" | "undeclared-resource" | "undeclared-host-path" | "decision-without-adr" | "action-not-probed" | "missing-receiver-calibration" | "cross-step-inconsistency" | "spec-wide-scope-violation" | "...",
  "severity": "block" | "warn" | "info",
  "location": {
    "scope": "step" | "cross-step" | "spec-wide",
    "step": 3,                      // present when scope=step
    "steps": [3, 5],                // present when scope=cross-step
    "ref": "action" | "verification" | "why" | "resources" | "first-principles" | "receiver-calibration" | null
  },
  "message": "<one-line, human-readable, ≤140 chars>",
  "suggested_fix": "<optional one-line, ≤140 chars>",
  "dismissable": true | false       // false = block-severity findings cannot be dismissed; true = user may accept-risk via signed dismissal
}
```

**Accept-risk path for Tier 3 false positives:** the user can dismiss a `dismissable: true` finding by adding it to a `# tier3-dismissed:` block at the bottom of the spec, paired with a one-line reason. Dismissals are recorded in the lock metadata (Decision 8 below) so future audits can see *what was dismissed and why*. `block`-severity findings from Tier 1/2 are `dismissable: false` and cannot be cargo-cult-ed away.

**Why JSON + structured locations:** the pragma-test-gaming-guard pattern. Narrative reviews from a same-family LLM can be papered over by the next turn's same-family LLM. Typed findings cannot. Even when DeepSeek reviews, the *output contract* is machine-checkable; downstream prompts can't relitigate the verdict. Without the explicit dismiss path (Copilot review 2026-05-05), users would cargo-cult edits just to clear the gate, defeating the purpose.

### 5. Block vs warn vs info

Every finding has `severity`. Only `block` halts the lock; `warn` and `info` are surfaced to the user but do not gate.

| Severity | Meaning | Examples |
|---|---|---|
| `block` | spec cannot lock until fixed | missing `why:` on any step; soft verification (`true`, `echo done`); action with NeverAutonomous verb but no explicit user-confirm note in §5 Physics Guardrails |
| `warn` | spec can lock but user should know | undeclared resource the inference would catch; decision marker without matching ADR; verification doesn't directly probe action effect (might still be valid via side-channel) |
| `info` | metadata for the human | spec is in scope of metis-boundary topics (per essay); marginal-spec-yield estimator suggests spec is over-specified |

**Why three levels:** Tier 1 produces mostly `block` (deterministic violations). Tier 2 mostly `warn` (heuristic gaps). Tier 3 mostly `warn`/`info` (judgment calls, not contractual violations). Severity calibration prevents Tier 3 from blocking on opinion.

### 6. Receiver-calibration is a first-class spec field — split hard contract from human notes

New `## 8. Receiver Calibration` section in `specs/template.spec.md`, structured into **two subsections** with different semantics (Copilot review 2026-05-05: hard policy and fuzzy claims must not share a namespace, or the evaluator will produce noisy false positives on the fuzzy half).

```markdown
## 8. Receiver Calibration

### 8.1 Hard contract (machine-enforced — `block` severity on violation)

- mutates: /opt/btc-poller/, /etc/systemd/system/, /var/log/btc-prices/, TCP 9100
- never-touches: /home, /root, /etc/passwd, /etc/shadow, network ports outside :9100
- decision-budget: 1 paid API call per minute (CoinGecko free tier)
- reboot-survival: required | best-effort | none

### 8.2 Human-facing notes (informational only — `info` severity)

- assumes: knows-systemd, knows-python-stdlib, knows-coingecko-api
- runtime-flavor: A8 (Debian 13, Ryzen 7 8745HS, 32GB RAM)
- expected-author-skill: senior backend engineer
```

**§8.1 enforcement:** every action's path is in `mutates`; no action touches `never-touches`; ports/quotas declared elsewhere in the spec match. Mismatches are `block`. Cross-check is mechanical — Tier 1 territory, no LLM judgment needed.

**§8.2 enforcement:** none, by design. This block is for the *human* reviewing the draft, not the evaluator. The evaluator may surface §8.2 contents in its findings as `info` severity but never blocks on them. Trying to machine-check `assumes: knows-systemd` produces noise (an action involving `systemctl` either matches or doesn't — what does "knows-systemd" really mean to a regex?). Keep the fuzzy half human-only.

**The §8.1 block is the spec author's contract with the executor**, per the essay's "specs include explicit precedence ordering on conflicting priors and explicit escalation paths when no precedence applies." §8.2 is the metadata that helps a *human* reviewer calibrate their expectations of the spec, not a machine.

### 7. CDLC mapping (where each phase lives)

| CDLC phase | Where it lives in Spectre |

v0.3 ships exactly Evaluate. The other phases are intentionally not in scope and are listed here so future planning can find them.

### 8. Lock metadata records evaluator version + policy hash for reproducibility

When a spec locks, the evaluator writes a `.eval.json` sidecar at `specs/<slug>.spec.md.eval.json` recording:

```json
{
  "evaluator_version": "0.3.0",
  "tiers_run": [1, 2, 3],
  "policy_hash": "<sha256 of effective severity policy at lock time>",
  "config_path": "~/.spectre/reviewer.toml",
  "config_hash": "<sha256 of config file contents at lock time>",
  "findings_summary": {
    "block_count": 0,
    "warn_count": 2,
    "info_count": 5,
    "dismissed_t3_count": 1
  },
  "dismissals": [
    {"finding_kind": "...", "step": 3, "reason": "..."}
  ],
  "deepseek_model_version": "deepseek-v4-pro@2026-05-05",
  "locked_at": "2026-05-05T16:00:00Z"
}
```

**Why this exists (Copilot review 2026-05-05):** without it, the same `.spec.md` could lock under different policies on different machines and no audit could later distinguish "this spec was evaluated under strict v0.3.2 policy" from "this spec was evaluated under permissive v0.3.0 policy with all warnings dismissed." The `.eval.json` makes lock decisions reproducible-by-inspection.

**No weaker-local-overrides rule:** `~/.spectre/reviewer.toml` MAY raise severity (warn → block) but MUST NOT lower it (block → warn). The evaluator validates this on config load and refuses to run with a downgrade-config. Strictness is one-way.

### 9. Out of scope (explicit, with reasoning)

| Out of scope | Why | Where it goes |
|---|---|---|
| **Test-gaming defense** (Pragma-style architecture applied to `/implement`'s test outputs) | Different attack surface (test→code coverage, not spec→action completeness). User decision: separate plan, separate version. | **v0.4** |
| **Auto-fix from findings** (evaluator proposes a refined draft) | Author/reviewer separation: the reviewer must not become the author. | Possibly v0.5 with strict provenance trail. |
| **Cross-spec evaluator** (consistency between specs/.graph.md and a new draft) | Plan A scope; needs the graph traversal Plan A already wired. | Could land in v0.3 as Plan B if scope allows; likely v0.5. |
| **Auto-rerun on spec edit** (file watcher → evaluator) | Violates `no manual invocation` rule the wrong way: re-running on every keystroke is noise, not signal. | Eval fires on `/vision refine` and on each `yes/refine/cancel` decision point. |
| **Distribution / registry of evaluator findings** (share findings across team) | Single-user (jounes) doesn't need it. | v0.5+ if multi-user lands. |
| **Local LLM as Tier 3** | User policy: no local LLM when away from home (thermal risk on A8's Radeon 780M; documented incident 2026-04-11). | DeepSeek v4 Pro is the locked choice. |

## Open questions (to be resolved before Plan A implementation)

1. **Tier 1 finding count cap.** A truly-bad spec might produce 50 findings. Do we cap at 10 (force user to fix the worst, re-run) or stream all? Cap = forced prioritization but loses signal. No-cap = noise risk. Probably cap at 20 with `--all` override.
2. **Tier 3 cost ceiling.** A long spec could trigger multi-call DeepSeek probing. Hard ceiling on tokens-per-spec? On dollars-per-month? Both? `~/.spectre/reviewer.toml` budget enforcement.
3. **Receiver-calibration discoverability.** If §8 is absent from a spec, is that `block` (must be present), `warn` (recommended), or `info` (optional)? Recommend `warn` initially; promote to `block` after 30 days of corpus data.
4. **Findings as ADR substrate.** When a Tier 3 finding causes the user to refine, should the *resolution* be auto-captured as an ADR ("considered X, rejected because Y per evaluator")? Probably yes, but the trigger (which findings → ADR) needs definition.
5. **Plan B / C scope.** v0.3 might carry a Plan B (cross-spec consistency, evaluator-driven graph-staleness detection) and a Plan C (Marginal Spec Yield estimator — quantitative stopping rule). Decide after Plan A ships and the corpus exists to inform scope.

## Success criteria (binary)

1. `/vision` for the same BTC poller spec surfaces **4 of 5** v0.2.2 failures as findings BEFORE the user confirms lock. The 5th (port 8765 collision with running daemon) requires live host-state probing, which is **out of scope for v0.3** and tracked in `docs/superpowers/ideas/post-v0.2.2-followups.md` item #4 — Copilot review 2026-05-05 corrected an overstatement here.
2. Tiers 1 + 2 (both default-on) together run in <2s on a 7-step spec.
3. Tier 3 runs in <30s on a 7-step spec via DeepSeek v4 Pro.
4. A spec with `severity: block` findings cannot be locked. The user must `refine`, `cancel`, or — for `dismissable: true` Tier 3 findings only — add a signed dismissal to the spec.
5. `~/.spectre/reviewer.toml` absent → Tiers 1+2 fire (deterministic, local). Tier 3 fires only when the config is present AND has explicit `tier3.enabled = true`.
6. The pragma-test-gaming-guard pattern applies: a follow-up `/vision refine` cannot relitigate a `dismissable: false` finding via narrative — only by structurally addressing the JSON `kind` and `location`.
7. JSON findings are stable across two runs of Tier 1+2 on the same spec text (deterministic).
8. The receiver-calibration §8.1 hard contract is cross-checked against actions' actual path captures; mismatches block lock. §8.2 human-facing notes are surfaced as `info` and never block.
9. Every locked spec has a sibling `.eval.json` sidecar with evaluator version, policy hash, tier list, dismissals, and DeepSeek model version (when applicable). A spec with no sidecar is detectable post-hoc as "locked under a pre-v0.3 evaluator or under no evaluator at all."
10. `~/.spectre/reviewer.toml` cannot weaken severity (block → warn). The evaluator refuses to load a downgrade-config and exits with a clear error.

## Peer Review Notes (Copilot/GPT-5.4)

Reviewed 2026-05-05 by GPT-5.4 via `gh copilot`. All six findings acted on; one finding partially deferred:

- **#1 (pipeline ordering):** Adopted. Decision 1 rewritten to introduce the *review bundle* concept — preview ADRs and preview Resource nodes are materialized before the evaluator runs, so Tier 1's "decision-without-ADR" and "undeclared-resource" checks have something to validate. Bundle commits to disk only after the user replies `yes`.
- **#2 (Tier 2 should be default-on):** Adopted. Tier 2 is now baseline alongside Tier 1; only Tier 3 (paid DeepSeek call) remains opt-in. Reflected in Decision 2 default-tier-policy block, success criteria #2 and #5.
- **#3 (port-collision claim overstated):** Adopted. Success criterion #1 corrected from "all 5 v0.2.2 failures" to "4 of 5" — the port-already-held-by-rule-router case requires live host probing which is genuinely out of v0.3 scope. Tracked separately in ideas-doc item #4.
- **#4 (lock metadata reproducibility):** Adopted. New Decision 8 — `.eval.json` sidecar records evaluator version, policy hash, tiers run, model version, dismissals. Plus a no-weaker-local-overrides rule on the config loader.
- **#5 (Receiver Calibration mixes hard contract and fuzzy notes):** Adopted. Decision 6 split into §8.1 (hard contract, machine-enforced, `block` severity) and §8.2 (human-facing notes, `info` only). Trying to machine-check `assumes: knows-systemd` was the noise risk Copilot named.
- **#6 (finding schema too step-bound + no accept-risk path):** Adopted. Decision 4 rewritten — `location` now has explicit `scope: step | cross-step | spec-wide`, plus a `dismissable` flag. - **Simpler alternative (review bundle as primary architecture):** Adopted as the central framing of Decision 1. The bundle approach was structurally cleaner than the original "evaluator runs after ADR write" framing.

Nothing skipped. The review caught real architectural issues; updated brief is materially better.
