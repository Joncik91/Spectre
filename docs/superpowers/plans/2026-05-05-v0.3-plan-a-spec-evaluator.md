# v0.3 Plan A — Pre-Lock Spec Evaluator Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship the pre-lock spec evaluator described in `docs/superpowers/specs/2026-05-05-spectre-v0.3-spec-evaluator.md`. Three tiers (AST classifier, coverage gate, DeepSeek judge), JSON findings with structured locations + dismiss path, review-bundle architecture, `.eval.json` lock-metadata sidecar, hard/soft Receiver Calibration split. Ends in `v0.3.0`.

**Architecture:** Six new stdlib-only modules (`bin/findings.py`, `bin/spec_ast.py`, `bin/coverage_gate.py`, `bin/llm_judge.py`, `bin/spec_evaluator.py`, `bin/eval_metadata.py`) plus modified `skills/vision/SKILL.md`, `specs/template.spec.md`, and a new user config at `~/.spectre/reviewer.toml`. Evaluator fires after `/vision` Step 6 (draft-to-disk) and before §6.5 (ADR write), operating on a transient review bundle that materializes preview ADRs / preview Resource nodes / preview tier classifications without committing them. Tiers 1+2 are baseline (always run, local, deterministic); Tier 3 is opt-in via the config file. Findings are typed JSON with `scope: step | cross-step | spec-wide` locations and a `dismissable` flag. `block`-severity findings halt lock; `dismissable: true` Tier 3 findings can be dismissed via a signed `# tier3-dismissed:` block in the spec, recorded in the lock-metadata sidecar.

**Tech Stack:** Python 3.11+ (stdlib only). `re`, `pathlib`, `json`, `hashlib`, `tempfile`, `urllib.request`, `dataclasses`, `tomllib` (3.11+ stdlib) already in scope. `pytest` for tests. No new dependencies. DeepSeek v4 Pro accessed via `urllib.request` — no `openai` SDK.

class FindingLocation:
    scope: str  # "step" | "cross-step" | "spec-wide"
    step: int | None = None
    steps: list[int] | None = None
    ref: str | None = None  # "action" | "verification" | "why" | "resources" | "first-principles" | "receiver-calibration"

```python
"""Finding dataclass + JSON schema for the v0.3 spec evaluator. Stdlib only."""
from dataclasses import dataclass, field, asdict
from typing import Any

KNOWN_KINDS = {
    "missing-why",
    "soft-verification",
    "undeclared-resource",
    "undeclared-host-path",
    "decision-without-adr",
    "action-not-probed",
    "missing-receiver-calibration",
    "cross-step-inconsistency",
    "spec-wide-scope-violation",
    "calibration-hard-violation",
    "tier3-context-gap",
    "tier3-attacker-view",
    "tier3-spec-asserts-wrong",
}

SEVERITIES = {"block", "warn", "info"}
SEVERITY_ORDER = {"info": 0, "warn": 1, "block": 2}
SCOPES = {"step", "cross-step", "spec-wide"}
MAX_MESSAGE_LEN = 140
MAX_FIX_LEN = 140


@dataclass
class FindingLocation:
    scope: str
    step: int | None = None
    steps: list[int] | None = None
    ref: str | None = None

    def __post_init__(self) -> None:
        if self.scope not in SCOPES:
            raise ValueError(f"unknown scope: {self.scope!r}")


@dataclass
class Finding:
    tier: int
    kind: str
    severity: str
    location: FindingLocation
    message: str
    suggested_fix: str | None = None
    dismissable: bool = False

    def __post_init__(self) -> None:
        if self.kind not in KNOWN_KINDS:
            raise ValueError(f"unknown finding kind: {self.kind!r}")
        if self.severity not in SEVERITIES:
            raise ValueError(f"unknown severity: {self.severity!r}")
        if self.tier not in {1, 2, 3}:
            raise ValueError(f"unknown tier: {self.tier!r}")
        if len(self.message) > MAX_MESSAGE_LEN:
            raise ValueError(f"message exceeds 140 chars: {len(self.message)}")
        if self.suggested_fix is not None and len(self.suggested_fix) > MAX_FIX_LEN:
            raise ValueError(f"suggested_fix exceeds 140 chars: {len(self.suggested_fix)}")


def to_dict(f: Finding) -> dict[str, Any]:
    return asdict(f)


def from_dict(d: dict[str, Any]) -> Finding:
    loc_data = d["location"]
    loc = FindingLocation(
        scope=loc_data["scope"],
        step=loc_data.get("step"),
        steps=loc_data.get("steps"),
        ref=loc_data.get("ref"),
    )
    return Finding(
        tier=d["tier"],
        kind=d["kind"],
        severity=d["severity"],
        location=loc,
        message=d["message"],
        suggested_fix=d.get("suggested_fix"),
        dismissable=d.get("dismissable", False),
    )


def max_severity(fs: list[Finding]) -> str:
    if not fs:
        return "info"
    return max(fs, key=lambda f: SEVERITY_ORDER[f.severity]).severity
```

- [ ] **Step 4: Implement `bin/spec_ast.py`**

Top-level signature:

```python
def classify(spec_path: Path, *, preview_adrs: list[str] | None = None) -> list[Finding]:
    """Tier 1 deterministic classifier. Returns Finding list (possibly empty)."""
```

Implementation parses the spec markdown, extracts `## 6. Steps` YAML block, walks each step. For each check in the closed list, run the corresponding detector. Return all findings. Use `bin.resources.extract_resources_from_action` for path captures and `bin.tier.classify` for tier verdicts (both already exist from v0.2.x).

Specifically:
- `missing-why`: walk YAML, any step without `why` key → `Finding(kind="missing-why", severity="block", location=FindingLocation(scope="step", step=N, ref="why"))`.
- `soft-verification`: regex check on each step's `verification:` against `^(true|echo .+|:)$` and similar.
- `action-not-probed`: extract paths from `action`, check each appears in `verification`. Heuristic — false-positive-prone, hence `warn` not `block`.
- `decision-without-adr`: scan §2 for decision markers (case-insensitive `decision:`, `we choose .* because`, `.* over .* because`), cross-check against `preview_adrs` parameter.
- `missing-receiver-calibration`: check §8.1 fields present.
- `calibration-hard-violation`: union of all path captures across steps; intersect with `never-touches:`; verify subset of `mutates:`.

**Tier 2 checks:**

1. **Resource coverage** — for each step's action, run `bin.resources.extract_resources_from_action`. Every inferred Resource MUST appear in that step's `resources:` declaration. Mismatch → `Finding(kind="undeclared-resource", severity="warn")` (warn because the inference is heuristic — see ideas-doc #4 for the heredoc/PORT-constant gap).
2. **Host-path coverage** — `bin.tier.classify` returns reasons including `path 'X' → host`. Every host-tier path MUST appear in §8.1 `mutates:`. Mismatch → `Finding(kind="undeclared-host-path", severity="block")`.
3. **Action-effect probe** — for actions that write a file, the verification MUST contain a structural check on that file (e.g. `[ -f <path> ]`, `grep ... <path>`, `python3 -c '...open("<path>")...'`). Stronger than Tier 1's text-match heuristic. `warn` (not `block` — some actions have side-channel effects that probe legitimately without the path appearing).

**3. Type consistency:** `FindingLocation`, `Finding`, `ReviewBundle`, `EvaluatorResult` shapes are consistent across Tasks 2, 8, and the skill prose in Task 10. ✓

**4. Risks the implementer should know:**

- **Tier 1's `action-not-probed` heuristic is regex-based and prone to false positives.** That's why it's `warn`, not `block`. Tier 2 has the proper coverage gate. The two are complementary, not redundant — Tier 1 is fast and produces fast feedback in the IDE; Tier 2 is structurally complete but slower. Both are default-on.

- **The DeepSeek API key handling.** The config stores the *name* of the env var, not the key. The key lives only in env at runtime. Never log the key. Never check the key into git. The `~/.spectre/reviewer.toml` MAY be committed if it points to env vars, but `.spectre/reviewer.toml.example` in the repo MUST NOT contain a real key.

- **The `.eval.json` sidecar must be atomic-written.** Use the existing `bin._scratchpad.atomic_write` pattern, not direct file writes. A torn sidecar means the next `/implement` reads an inconsistent eval state and may halt or false-pass.

- **The §8.1 hard contract cross-check is dependent on `bin.tier.classify` and `bin.resources.extract_resources_from_action` being correct.** Item #5 in `docs/superpowers/ideas/post-v0.2.2-followups.md` (now superseded by v0.3 in spirit) names known tier classifier holes (bare `systemctl`, `.service` as repo). The evaluator catches what those miss at draft time, but if either tool returns wrong path captures, the §8.1 check inherits the wrong-ness. Task 4's `calibration-hard-violation` check should default to `block` only when path captures are non-empty; an empty capture set means "tier classifier produced no host-path reasons" and the check is non-applicable, not "spec is safe."

- **Tier 3 latency is real.** 30s budget. If DeepSeek is slow or rate-limited, Tier 3 returns an `info`-severity sentinel finding ("Tier 3 unavailable") and the evaluator proceeds with Tier 1+2 results. Hard fail on Tier 3 unavailability would block all locks when the user is offline — wrong UX.

- **The dismiss path is a footgun if abused.** `# tier3-dismissed:` blocks should be reviewed in PR. The `.eval.json` sidecar records each dismissal with reason; CI on the user's repo can lint for "spec locked with N dismissals" and fail if N > threshold.
